# Minion Toolkit v2.3.0 — Octopus-Inspired Improvements

**Date:** 2026-03-08
**Status:** Approved
**Inspiration:** [claude-octopus](https://github.com/nyldn/claude-octopus) — multi-model orchestration patterns adapted for single-model parallel execution.

## Overview

Four features inspired by claude-octopus, adapted to Minion Toolkit's architecture:

| Feature | Origin Idea | Our Adaptation |
|---------|-------------|----------------|
| Stall Detection | Loop stall detection in `/octo:loop` | Git SHA comparison in cycle transitions |
| Smart Workflow Selection | Intent-based smart routing in `/octo` | Complexity heuristic auto-selects workflow |
| Intent Capture | Session intent contracts | `.minion/intent.md` on first run |
| Pre-Built Agents + Phase Roles | 10 droids with phase-specific personas | Layered prompts: domain agents + role overlays |

## Feature 1: Stall Detection in Cycles

### Problem
Review→fix→review cycles always run all iterations even if the fix phase produces zero changes, wasting time and cost.

### Solution
After a fix worker reports back, compare `git rev-parse HEAD` before and after. If the SHA is unchanged → no commit was made → stall detected → exit cycle early.

### Where
Orchestrator Step 7 (phase transition), Case B/C — cycle target phase completed with non-success status.

### Logic
```
Before spawning fix worker:
  capture: pre_fix_sha = git rev-parse {branch_name}

After fix worker reports back:
  capture: post_fix_sha = git rev-parse {branch_name}

  if pre_fix_sha == post_fix_sha AND status != "success":
    → Log: "Stall detected — fix phase produced no changes. Exiting cycle early."
    → Mark task with current status (e.g., review_failed)
    → Skip re-review, continue to next phase or end task
  else:
    → Normal cycle behavior (loop back, increment cycle_count)
```

### Edge Cases
- Fix worker reports `success` with no changes → valid (review found no issues), not a stall
- Fix worker reports `stuck`/`blocked` → already handled by existing logic, stall check is bonus safety

### Changes
- `commands/minion.md`: ~8 lines in Step 7, Case B/C

## Feature 2: Smart Workflow Auto-Selection

### Problem
Users must remember `--workflow tdd` vs `--workflow quick` vs `--workflow secure`.

### Solution
New Step 1.2 between task parsing and workflow resolution. Scans all parsed tasks and scores workflows using a complexity heuristic.

### Where
New Step 1.2 — only runs when no `--workflow` flag is provided.

### Selection Rules (first match wins)
```
1. has_security_keywords                                          → "secure"
2. task_count == 1 AND total_files <= 3 AND NOT has_dependencies  → "quick"
3. has_test_keywords OR has_dependencies OR task_count >= 3       → "tdd"
4. default                                                        → "default"
```

### Keywords
- Security: "security", "auth", "OWASP", "vulnerability", "encryption", "permission"
- Test/TDD: "test", "TDD", "spec", "coverage"

### Priority
```
Explicit --workflow flag  →  always wins (skip auto-selection)
Auto-selected             →  shown with "(auto-selected — reason)" in confirmation
User can override         →  via "Adjust" option in Step 3
```

### Changes
- `commands/minion.md`: ~25 lines, new Step 1.2

## Feature 3: Intent Capture

### Problem
Workers have no "north star" for what the overall run is trying to achieve. Review phases can't validate against the original intent.

### Solution
New Step 0 — ask the user 2 questions on fresh runs, save to `.minion/intent.md`.

### Where
New Step 0 — before parsing tasks. Skipped on `--resume`, `--dry-run`, or if `.minion/intent.md` already exists.

### Flow
```
Step 0: Capture Intent

  1. Ask: "What's the goal of this run?"
     → store as `goal`

  2. Ask: "What does success look like? (How will you know it's done?)"
     → store as `success_criteria`

  3. Write .minion/intent.md:
     # Run Intent

     ## Goal
     {goal}

     ## Success Criteria
     {success_criteria}

     ## Tasks File
     {tasks_file_path}

     ## Timestamp
     {ISO timestamp}
```

### How Workers Use It
1. **Review phases** — append to `PHASE PROMPT`: "Reference `.minion/intent.md` for the overall run goal and success criteria when evaluating this implementation."
2. **Post-run report** — Step 8 includes "Intent Alignment" section:
   ```
   ## Intent Alignment
   - Goal: {goal}
   - Success Criteria: {success_criteria}
   - Assessment: {based on task outcomes}
   ```

### Skip Conditions
- `--resume` → intent already captured
- `.minion/intent.md` exists → don't re-ask
- `--dry-run` → skip (preview only)

### Changes
- `commands/minion.md`: ~20 lines for Step 0, ~5 lines in Step 6 (worker prompt), ~8 lines in Step 8 (report)

## Feature 4: Pre-Built Agent Library + Phase-Aware Selection

### Problem
Only `minion-worker` and `security-reviewer` ship. No phase-aware persona switching. Auto-detection rules are hardcoded in the orchestrator.

### Solution
Layered prompt composition: domain agents (what they know) + role overlays (how they behave per phase).

```
Final worker prompt = base minion-worker rules
                    + domain agent persona (if matched)
                    + role overlay (if phase has one)
                    + task-specific context
```

### Domain Agents (new `agents/*.md` files)

| Agent | Matches On |
|-------|-----------|
| `backend-architect` | `.service.ts`, `.controller.ts`, `.entity.ts`, `.dto.ts`, `.module.ts`, `services/`, `controllers/`, keywords: "endpoint", "API", "database", "migration", "NestJS" |
| `frontend-architect` | `.tsx`, `.jsx`, `.css`, `.scss`, `components/`, `pages/`, `hooks/`, keywords: "component", "UI", "form", "widget", "React" |
| `devops-engineer` | `Dockerfile`, `.yml`, `.yaml`, `ci/`, `.github/`, keywords: "deploy", "CI/CD", "pipeline", "Docker", "infrastructure" |
| `data-engineer` | `.sql`, `migrations/`, `seeds/`, keywords: "migration", "schema", "query", "ETL" |
| `tech-writer` | `.md`, `docs/`, keywords: "documentation", "README", "ADR", "changelog" |

Each agent is ~30 lines with YAML frontmatter containing a `matches` field:

```yaml
---
name: backend-architect
description: API, service, and database architecture specialist
matches:
  extensions: [".service.ts", ".controller.ts", ".entity.ts"]
  paths: ["services/", "controllers/", "modules/"]
  keywords: ["endpoint", "API", "database", "migration"]
---

# Persona: Backend Architect
{domain expertise, priorities, focus areas}
```

### Role Overlays (new `roles/*.md` files)

| Role | Used In Phases |
|------|---------------|
| `researcher` | `plan` |
| `tdd-developer` | `implement` |
| `code-reviewer` | `review` |
| `security-auditor` | `security-review` |
| `performance-reviewer` | `review` (when task mentions perf keywords) |

Each overlay is ~20 lines — behavioral prompt that modifies HOW the worker approaches its task.

### Phase-to-Role Mapping

New optional `Role:` field in workflow phase definitions:

```markdown
## Phase: plan
- Prompt: "Analyze and plan: {task}"
- Role: researcher
- Agent: minion-worker
- Gate: artifact
```

If `Role:` is not specified, no overlay is applied (backward compatible).

### Updated Selection Algorithm (Step 1.7)

```
For each task, for each phase:

  1. Explicit Agent: field in task?
     → use that agent (unchanged)

  2. Phase has Role: field?
     → load role overlay from roles/{role}.md

  3. Auto-detect domain agent:
     a. Read agent .md files, extract `matches` from frontmatter
     b. Score: extension match = 3, path match = 2, keyword match = 1
     c. Highest score wins
     d. Ties → project-level preferred, then alphabetical
     e. No match → minion-worker (fallback)

  4. Compose prompt: domain persona + role overlay + task context
```

### Worker Prompt Additions

Two new optional blocks in Step 6 (spawn workers):

```
DOMAIN PERSONA: {contents of matched domain agent's persona section}
ROLE OVERLAY: {contents of phase's role overlay}
```

### Existing Agents Unaffected
`cloudx-backend`, `cloudx-frontend`, `code-quality`, `security-reviewer` stay as-is. They're project-specific and score higher due to project-level priority.

### Changes
- 5 new domain agent files: `agents/{backend-architect,frontend-architect,devops-engineer,data-engineer,tech-writer}.md`
- 5 new role overlay files: `roles/{researcher,tdd-developer,code-reviewer,security-auditor,performance-reviewer}.md`
- `commands/minion.md`: Step 1.7 rewrite (~40 lines) for scored matching + Step 6 update (~10 lines) for layered prompt composition
- `workflows/{tdd,full-pipeline,secure}.md`: Add `- Role:` field to relevant phases
